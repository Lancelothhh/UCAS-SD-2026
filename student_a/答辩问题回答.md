# 答辩问题回答（核心算法 · 成员 A）

> 本文档记录 4 个答辩必问问题的标准回答，回答均贴合 `src/avl_tree.c` 的实际实现。
>
> **重要约定：** 本项目平衡因子定义为 **`balance_factor = 左子树高度 − 右子树高度`**
> （见 `src/avl_tree.c:18`）。因此「**正数 = 左边高，负数 = 右边高**」。
> 答辩时务必先说清这一点，否则 LL / RR 会讲反。

## 问题 1：为什么 AVL 树查找、插入、删除是 O(log n)？

**一句话：** AVL 树始终保证树高 h = O(log n)，而这三种操作的代价都正比于树高。

- 查找、插入、删除本质上都是**从根往下走一条路径**，每层只做常数工作，所以单趟代价是 O(h)。
- AVL 的平衡条件——每个节点左右子树高度差 ≤ 1（代码中 `validate_rec` 第 230 行
  `abs(left_height - right_height) > 1` 即检查此条件）——保证了 h 最坏也只有 O(log n)。
- **证明思路（可能被追问）：** 设高度为 h 的 AVL 树**最少**节点数为 N(h)，则
  `N(h) = N(h-1) + N(h-2) + 1`，这是斐波那契递推，N(h) 随 h 指数增长；
  反推得 `h ≤ 1.44 · log₂(n)`，即 h = O(log n)。
- 插入 / 删除虽然多了「回溯 + 旋转」，但单次旋转是 O(1)，回溯路径长度也是 O(h)，
  所以总代价仍是 O(log n)。

## 问题 2：插入后如何根据平衡因子判断 LL、RR、LR、RL？

**对应代码：** `rebalance()`（`src/avl_tree.c:68–94`）。先 `update_height`，再算当前节点 bf，分两大类：

- **bf > 1（左边过高）**，再看**左孩子**的 bf：
  - 左孩子 bf ≥ 0 → **LL 型** → 直接对当前节点**右旋** `rotate_right`（第 83 行）
  - 左孩子 bf < 0 → **LR 型** → 先对左孩子**左旋**，再对当前节点**右旋**（第 80–83 行）
- **bf < −1（右边过高）**，再看**右孩子**的 bf：
  - 右孩子 bf ≤ 0 → **RR 型** → 直接对当前节点**左旋** `rotate_left`（第 90 行）
  - 右孩子 bf > 0 → **RL 型** → 先对右孩子**右旋**，再对当前节点**左旋**（第 87–90 行）

**记忆口诀：** 外侧失衡（LL/RR）转一次；内侧失衡（LR/RL）先把内侧拐点转直、变成外侧，再转一次。
判断「内 / 外」的方法：看「当前节点 bf 符号」与「对应孩子 bf 符号」是否一致——
一致是外侧（LL/RR），相反是内侧（LR/RL）。

### 四种情况速查表

| 类型 | 当前节点 bf | 失衡孩子 bf | 处理 |
|------|-----------|-----------|------|
| LL | > 1  | 左孩子 ≥ 0 | 右旋 |
| LR | > 1  | 左孩子 < 0 | 左孩子先左旋，再右旋 |
| RR | < −1 | 右孩子 ≤ 0 | 左旋 |
| RL | < −1 | 右孩子 > 0 | 右孩子先右旋，再左旋 |

## 问题 3：删除非叶子节点时，如何用后继节点替代，再继续往下删？

**对应代码：** `delete_rec()`（`src/avl_tree.c:124–151`）。删除分三种情况：

1. **叶子 / 只有一个孩子**（第 138–141 行）：直接用那个孩子（或 `NULL`）接上去，`free` 掉自己。
2. **有两个孩子**（第 142–147 行）：不能直接删，否则树会断。做法是：
   - 找**后继节点** = 右子树里的最小节点 `min_value_node(root->right)`
     （一路向左到底，第 115–122 行）；
   - 把后继的 key **复制**到当前节点（第 145 行 `root->key = successor->key`）
     —— 注意是**值替换，不动指针结构**；
   - 然后转化成「在右子树里删除那个后继 key」的子问题，
     递归 `delete_rec(root->right, successor->key, ...)`（第 146 行）。

**关键点：** 后继节点在右子树里是「最小」的，所以它**最多只有一个右孩子、没有左孩子**，
于是对它的递归删除一定落到情况 1，能干净地真正释放掉。
本质是「把难删的双孩子情况，转嫁成一个好删的节点」。

### 详解：双孩子删除的完整流程（重点）

有两个孩子时**不能直接删**，否则两个孩子没地方挂。办法分三步：

1. **找后继。** 用 `min_value_node` 找到**右子树里最小的节点**——它叫后继，是整棵树里
   「比当前值大的数中最小的那个」（`src/avl_tree.c:115`，就是从右孩子一路往左走到底）。
2. **搬值替换。** 把后继的值搬到当前节点（`root->key = successor->key`），
   当前节点的「内容」就被替换了——**只改值，不动指针结构**。
3. **递归删后继。** 现在右子树里多了一个重复值，再递归去右子树把那个后继节点删掉。

**为什么这一步不会无限递归（关键）：** 后继是右子树的最小值，所以它**没有左孩子**
（否则左孩子更小，它就不是最小了），最多只有一个右孩子。
因此递归删它时**必然落到情况 1（叶子 / 单孩子），不可能再是情况 2**，一步就真正释放掉，
递归到此终止。

> 易错点更正：这一步只会落到**情况 1**，不会落到情况 2——因为后继一定没有左孩子，
> 不可能是「双孩子」节点。

## 问题 4：删除后为什么也必须回溯更新高度并重新平衡？

**因为删除同样会改变子树高度，从而可能破坏祖先节点的平衡。**

- 删掉一个节点后，从删除点到根这条路径上每个祖先的高度都可能变小，平衡因子随之变化，
  可能出现 |bf| > 1。
- 所以 `delete_rec` 在递归返回时（第 150 行 `return rebalance(root)`），
  对路径上**每一个**祖先都重新 `update_height + rebalance`，和插入是同一套机制。
- **与插入的重要区别（易被追问）：**
  - 插入最多触发**一次**旋转就能恢复整棵树平衡；
  - 删除可能需要**沿路径多次旋转**——一次旋转修好了局部，但旋转会让该子树高度再降 1，
    又可能让更上层的祖先失衡，所以必须一路回溯到根、逐层检查，不能中途停。
- 本项目代码天然做对了这点：因为 `rebalance` 写在递归的**返回路径**上，每层都会执行，
  所以不管删除引发几次连锁失衡都能覆盖到。

### 详解：删除回溯 rebalance 的必要性（重点）

- **为什么要回溯：** 删掉一个节点会让**某一侧变矮**，导致祖先的平衡因子可能变成 ±2。
  所以和插入一样，递归 `return` 回来的路上，**每个祖先都执行 `rebalance(root)`**。
- **比插入更麻烦的一点：** 插入**修一次就够**，删除**可能要修好几次**。
  原因是一次旋转可能让该子树高度**又减 1**，把失衡**继续向上传递**，
  所以必须**一路修到根**，不能修一次就停。
- 代码里「**每层都 `rebalance`**」（`delete_rec` 第 150 行写在递归返回路径上）
  恰好覆盖了这种连锁失衡的情况——不需要任何特判，回溯天然逐层检查。

> 一句话对比：插入的旋转能把子树高度**还原**，所以止于一次；
> 删除的旋转会把子树高度**降低**，所以可能连锁、必须修到根。

## 补充：`static` 辅助函数为什么这样写？

`avl_tree.c` 里的 `max_int` / `node_height` / `rotate_left` 等都加了 `static`，
表示**内部链接**——只在本编译单元（`avl_tree.c`）可见：

- 避免与其它文件的同名符号冲突（如通用名 `max_int` 重复定义会导致链接错误）；
- 它们是**实现细节**，不属于 `avl_tree.h` 的公开 API，类似面向对象里的 `private` 辅助方法；
- 便于编译器**内联优化**这些小函数。

> 注意：这与「局部变量加 `static` 延长生命周期」是不同用法，只是复用了同一个关键字。

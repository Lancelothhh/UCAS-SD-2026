## 我的核心判断：答辩这 4 个问题是必问的

准备报告时，我认为下面 4 个问题是答辩老师**一定会问**的，必须在报告里把它们讲透，
不能只贴代码。它们正好覆盖了 AVL 树「为什么对、怎么平衡、怎么删、删完怎么收尾」的完整链条：

1. **为什么 AVL 树查找、插入、删除是 O(log n)？**
   —— 考的是对「平衡 ⇒ 树高 O(log n) ⇒ 操作 O(树高)」这条逻辑链的理解，
   要能引出斐波那契下界的证明。

2. **插入后如何根据平衡因子判断 LL、RR、LR、RL？**
   —— 考的是四种旋转的触发条件，是最容易被要求「现场画图推演」的点。

3. **删除非叶子节点时，如何用后继节点替代，再继续往下删？**
   —— 考的是双孩子删除的处理技巧（值替换 + 转化为删后继）。

4. **删除后为什么也必须回溯更新高度并重新平衡？**
   —— 考的是删除和插入的区别：删除可能连锁多次旋转，必须回溯到根。

## 为什么我特别强调这 4 个的顺序

我想按「正确性 → 平衡机制 → 删除难点 → 删除收尾」的顺序讲，是因为它们之间有递进关系：
- 先证明 O(log n)，立住「AVL 值得用」的地基；
- 再讲插入旋转，说明「平衡是怎么维持的」；
- 然后讲删除，因为删除比插入更难、更容易出错；
- 最后强调删除回溯，把「为什么删除要多做一步」这个易混点收尾。

## 我注意到 / 担心的细节

- **平衡因子方向容易讲反。** 我们代码里 bf = 左高 − 右高（`avl_tree.c:18`），
  正数代表左边高。答辩开场必须先声明这个约定，否则 LL/RR 一讲就错。
- **删除和插入的区别是采分点。** 插入只转一次，删除可能转多次——这个对比一定要主动讲出来，
  显得我真的理解了，而不是背代码。

## 延伸思考 1：有序数组「取中点建树」和 AVL 的关系

我一开始想：有一个从小到大的数据列，每次取中间值当节点，左侧建左子树、右侧建右子树，
反复递归，不就直接得到 AVL 了吗？

想清楚后的结论：**能得到一棵满足 AVL 平衡条件的 BST，但它只适用于静态场景，不能替代 AVL。**

- ✅ **静态建树**：对**已排好序**的数组取中点递归建树，得到高度 ⌈log₂(n+1)⌉ 的平衡 BST，
  每个节点左右高度差 ≤ 1，完全满足 AVL 不变量。这是 O(n) 一次性建树，比逐个插入
  （O(n log n)）更快。其本质就是**二分查找的递归结构**（也解释了折半查找为什么 O(log n)）。
- ❌ **不能替代 AVL**：取中点依赖「手里已有全部有序数据」。AVL 解决的是**动态**问题——
  数据一个一个来、还要随时增删。一旦插入新值，原中点就不再是中点，整棵树又失衡，
  而我不可能每来一个数据就全表重排、重建。

**一句话区分（答辩亮点）：** 取中点建树是静态有序表的最优一次性构造；
AVL 用旋转，是为了在动态增删中、不重建整棵树的前提下，把平衡「增量地」维持住。
两者目标不同，不是替代关系。

## 延伸思考 2：旋转的唯一目的和唯一约束

- **唯一目的：** 把失衡节点的平衡因子拉回 {−1, 0, 1}，降低过高一侧、抬升过矮一侧，
  从而把**树高维持在 O(log n)**。一句话——旋转只为维持平衡（控制树高）。
- **唯一约束：** 必须**保持 BST 中序序列不变**（左 < 根 < 右的有序性不能破坏）。

这就是旋转只能是那几种固定形态的原因：`rotate_right` / `rotate_left`
（`avl_tree.c:42–66`）里重新挂接子树 `t2`，正是为了改变树形的同时让中序遍历结果完全不变。

**记法：** 旋转 = 在「不改变中序顺序」这唯一约束下，做「降低树高」这唯一目的的最小局部调整。
验证旋转对不对的金标准：旋转前后中序遍历序列必须一模一样
（`validate_rec` 同时查 BST 序和高度差，就是在守这两条）。

## 延伸思考 3：merge 还有三条更快的路（按「好理解 → 最优」排）

**基线 = 现在的实现**：中序遍历 + 逐个 `avl_insert` 重插，**O(m·log(n+m))**。
慢在每插一个都要从根找一遍位置（log 的来源），m 个数就 m 次 log。

- **方法一：范围不重叠 → join，O(log n)。**
  若 T1 所有 key 都 < T2 所有 key，不用逐个插，直接利用两棵树的高度把矮的接到高的合适位置、
  再局部 rebalance，代价 O(|h₁−h₂|) = O(log n)。前提苛刻（范围完全不交错），
  一般用不上，只有「分裂后又拼回去」这类场景满足。

- **方法二：两棵差不多大 → 拉平 + 线性归并 + 取中点重建，O(n+m)。** ⭐ 最优雅、最好实现
  1. 中序遍历 T1 → 有序数组 A（O(n)）；
  2. 中序遍历 T2 → 有序数组 B（O(m)）；
  3. 像归并排序的合并那步，把 A、B 拉链式归并、顺手去重 → 有序数组 C（O(n+m)）；
  4. **对有序的 C 取中点递归建树** → 直接得到平衡 BST（O(n+m)）。

  > **和延伸思考 1 的呼应（重点）：** 第 4 步正是我之前想过的
  > [「静态有序表取中点一次性建树」](#延伸思考-1有序数组取中点建树和-avl-的关系)！
  > 当时的结论是「取中点只适合静态、不能替代动态 AVL」——这里恰好是它的正确用武之地：
  > 两棵 AVL 树**中序遍历天然有序**，所以合并退化成「两个有序数组归并」（O(n+m)、不需要任何
  > log），最后再用「取中点」把这个静态有序数组一次性重建成平衡树。
  > **动态结构（AVL）+ 静态重建（取中点）在这里合体了。**

- **方法三：理论最优 → 递归 Just Join union，Θ(m·log(n/m+1))。**
  分治（Blelloch–Ferizovic–Sun）：取 T1 的根 k，用 k 把 T2 `split` 成左右两半，
  递归 union 左右，再 `join` 回去。达到信息论下界、天然可并行；
  m≪n 时是 O(m·log(n/m))，m≈n 时是 O(n)。

### 怎么选：取决于两棵树的相对大小

| 场景 | 最佳方法 | 代价 |
|------|---------|------|
| 范围不重叠 | 方法一 join | O(log n) |
| 两棵差不多大（m ≈ n） | 方法二 拉平+归并+重建 | O(n+m) ≈ O(n) |
| 一棵远小于另一棵（m ≪ n） | 现在的逐个重插 / 方法三 | O(m·log n) / Θ(m·log(n/m+1)) |

**反直觉点：** 方法二虽然线性，但 m ≪ n 时反而更慢——光「拉平大树」就要 O(n)，
而这时直接把 m 个小数逐个插进去只要 O(m·log n) 更划算。
**没有一种方法在所有情况下都最快，要看 m 和 n 的比例。**

**答辩收口：** 现在的 O(m·log(n+m)) 不是最优——等大可线性归并+取中点重建做到 O(n+m)，
不重叠可 join 到 O(log n)，理论最优是 Just Join 的 Θ(m·log(n/m+1))；
我们选逐个重插是因为它**复用已验证的 `avl_insert`，正确性最易保证**，
且小树并入大树（m≪n）时性能也不差。

## 延伸思考 4：split 的实现与 O(log n) 优化

**现在的实现 = 遍历 + 分类 + 重插**（`avl_split` / `split_rec`，`src/avl_tree.c:341` / `:249`）：
中序遍历 `source`，`key <= x` 的插进 `less_equal`，`> x` 的插进 `greater`，两棵结果都用
`avl_insert` 建，天然平衡。`source` 是 `const`、原样不动；调用前两棵输出树要先 `avl_init`。

- **分界线：** `x` 本身归 `less_equal`（判断是 `<= x`），符合作业要求「一棵 ≤ x，另一棵 > x」。
- **中序非必须：** 分到哪棵只看 key 与 x 的大小，与遍历顺序无关，换前序/后序结果一样。
- **复杂度：** n 个节点各做一次 `avl_insert`（O(log n））→ **O(n·log n)**。
- **和 merge 的对称：** merge 是「两棵→一棵」，split 是「一棵→两棵」，现在都是「遍历 + 复用
  `avl_insert`」一个模子，正确性都极易证。

**更快的版本：递归 split + join，O(log n)。** 配一个 `join(L, k, R)`（已知 L < k < R，拼成
平衡 BST，代价 O(|h_L − h_R|) = O(log n)）：

```
// 返回 (L, R)：L = 所有 ≤ x，R = 所有 > x
split(node, x):
    if node == NULL: return (NULL, NULL)
    if x < node.key:                       // node.key > x，自己和右子树归 R 侧
        (L, R) = split(node.left, x)       // 只往左递归
        return (L, join(R, node.key, node.right))
    else:                                  // node.key <= x，自己和左子树归 L 侧
        (L, R) = split(node.right, x)      // 只往右递归
        return (join(node.left, node.key, L), R)
```

- **为什么是 O(log n)：** 每次只往一个方向递归 → 只走一条根到叶路径（O(log n)）；
  路径上每个节点做一次 join，单次代价是子树高度差，沿路径累加**错位相消（telescoping）**，
  总和仍 O(log n)。这是平衡树 split 的经典定理（Tarjan；Blelloch 等）。
- **项目为何不用：** 同 merge 的 trade-off——快版要实现 `join`（AVL 高度对齐 + rebalance），
  代码复杂易错；现在的「遍历 + 重插」复用已验证的 `avl_insert`，正确性一眼可见，作业够用。

**对偶关系（答辩亮点）：** 一般 merge(union) 的最优算法（Just Join）内部要调 **split**；
split 的最优算法内部要调 **join**。三者互为基石：`union → split → join`，
都建立在原子操作 `join` 之上。能讲出这层关系，体现的是对平衡树操作体系的整体理解。
